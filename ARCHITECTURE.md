# 🏗️ LifeOfPy Architecture

LifeOfPy is designed with a strict, decoupled, and highly modular architecture. This ensures that the engine is deterministic, reliable, and easily testable.

## High-Level Overview

### 1. The Core (`packages/core`)
The lowest level of the stack. Contains foundational primitives that the rest of the system relies on.
- **FileSystem Service**: A purely abstract layer for OS interactions, enforcing atomic writes and preventing partial state corruption.
- **Registry Provider**: Pluggable interfaces (GitHub, Local, S3) that fetch data without the rest of the engine knowing *where* it comes from.

### 2. The Engine (`packages/engine`)
The brain of LifeOfPy. It orchestrates the business logic of package management.
- **Manifest Validator**: A compiler-style diagnostic engine that strictly checks component schemas, dependencies, and licensing.
- **Registry Builder**: Transforms a folder of components into a compiled, deterministic, static JSON registry.
- **Dependency Graph**: Parses dependency trees, detects circular cycles using DFS, checks framework conflicts, and topologically sorts installation plans.
- **Downloader**: A transport-agnostic, queue-driven worker that fetches artifacts into a secure sandbox and validates their SHA-256 checksums.
- **Installer**: The final orchestrator. It uses transactional locks to atomically commit the validated staging area into the user's project space, managing history and rollback states.

### 3. The CLI
The developer interface. The CLI strictly consumes the Engine APIs. It handles terminal output (using Rich), argument parsing, and user prompts, but contains **zero** core business logic.

### 4. The Website
The public face of the component registry. A Next.js application that purely consumes the static JSON output generated by the Registry Builder. It does not own or mutate registry state.

## Design Philosophy
- **Determinism**: The same inputs must yield the exact same outputs (registry builds, installation plans).
- **Safety First**: Never mutate a user's project unless the entire transaction (download, verify, dependencies) has successfully passed in staging.
- **Decoupled IO**: Core logic should be testable without hitting the network or the disk.
